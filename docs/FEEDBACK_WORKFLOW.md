# マネぼう フィードバック運用フロー

## 🔄 フィードバック → 改善までの運用体制

---

## 📋 フロー全体

```
ユーザーからのフィードバック
       ↓
【ステップ1: 受付】
- GitHub Issues / Discussions に投稿
- サポートチームが確認
       ↓
【ステップ2: 分類】
- バグ / 機能提案 / 改善案 に分類
- カテゴリを選定（UI/UX/教育等）
       ↓
【ステップ3: 優先度付け】
- MoSCoW 方式で評価
- MUST / SHOULD / COULD / WON'T に分類
       ↓
【ステップ4: 担当・期限決定】
- 担当者をアサイン
- 対応期限を決定
       ↓
【ステップ5: 対応実施】
- コード修正 / ドキュメント作成
- テスト実施
       ↓
【ステップ6: 完了報告】
- GitHub で対応完了を通知
- CHANGELOG に記載
- ユーザーに返信
```

---

## 📌 ステップ別の定義

### ステップ1: 受付

**受け取り先:**
- GitHub Issues: バグ報告（テンプレート: BUG_REPORT_TEMPLATE.md）
- GitHub Discussions: 機能提案・質問
- メール: 個別相談

**確認項目:**
- [ ] フォーマットが正確か
- [ ] 既出でないか確認
- [ ] 環境情報が記載されているか

**対応**: 24 時間以内に「受け取り確認」をリプライ

**テンプレート:**
```
ご報告ありがとうございます。
確認させていただきますので、
以下の情報を追加していただけますか？

【質問1】...
【質問2】...
```

---

### ステップ2: 分類

**フィードバックタイプ:**

| タイプ | 説明 | 対応期限 |
|--------|------|---------|
| **🔴 Critical Bug** | アプリがクラッシュ・ログインできない | 24時間 |
| **🟠 High Priority Bug** | 機能が全く動作しない | 3日以内 |
| **🟡 Medium Bug** | 一部動作がおかしい | 1週間 |
| **🟢 Low Bug** | UI のバグ・軽微な問題 | 2週間 |
| **💡 Feature Request** | 新しい機能提案 | v1.1.0 以降検討 |
| **📝 Improvement** | 既存機能の改善 | 対応優先度による |

**分類方法:**
- コメントで `type: bug`, `type: feature`, `type: improvement` ラベルを付与
- 重要度ラベルを付与：`priority: critical`, `priority: high` 等

---

### ステップ3: 優先度付け

**MoSCoW 方式:**

| 優先度 | 基準 | v1.0.x | v1.1.0 |
|--------|------|---------|---------|
| **MUST** | システム動作に必須 | 即対応 | v1.0.1 で対応 |
| **SHOULD** | ユーザー体験を大幅改善 | 1-2週間 | v1.1.0 に含める |
| **COULD** | あると良い改善 | 今後検討 | SHOULD の後 |
| **WON'T** | スコープ外 | 見送り | 理由を記載して クローズ |

**評価基準:**
- 影響ユーザー数
- ビジネスインパクト
- 実装コスト
- 緊急度

---

### ステップ4: 担当・期限決定

**担当者アサイン:**

```
Assignee: @developer-name
Labels: type:bug, priority:high, category:UI
Milestone: v1.0.1 (リリース予定版)
Due Date: 2026-07-15
```

**期限設定:**

| 優先度 | 期限 |
|--------|------|
| Critical | 24 時間 |
| High | 3 日以内 |
| Medium | 1 週間以内 |
| Low | 2 週間以内 |

---

### ステップ5: 対応実施

**タスク:**

1. ローカル環境で再現確認
2. 原因特定
3. コード修正（機能提案の場合は設計）
4. ユニットテスト追加
5. ドキュメント更新
6. Git コミット

**Git コミットメッセージ形式:**
```
fix: [Issue#123] ログイン後のルーティングエラーを修正

Closes #123
```

---

### ステップ6: 完了報告

**GitHub 上での報告:**

```markdown
## 対応完了

以下の対応が完了しました：

- [x] コード修正
- [x] テスト実施（パス）
- [x] ドキュメント更新
- [x] デプロイ（Vercel）

**修正内容:**
[簡潔な説明]

**確認方法:**
[ユーザーが確認できる方法]

ご報告ありがとうございました。
本番環境に反映済みです。
```

**ユーザーへのリプライ:**

- GitHub で Close コメント
- CHANGELOG に記載
- 次のリリースノートで周知

## 📊 統計・定期報告

### 週次報告（毎週金曜）

```markdown
## 週次フィードバック統計

- 新規報告: X 件
- 対応完了: Y 件
- 対応中: Z 件
- 平均対応時間: XX 時間
- 満足度: ★★★★☆

- ...
```

## 👥 役割分担

| 役職 | 責任 |
|------|------|
| **ユーザー** | フィードバック投稿 |
| **サポート** | 受付・分類・優先度付け |
| **開発者** | 対応実施・テスト |
| **管理者** | フロー監視・期限管理 |

---

## 🎯 SLA（サービスレベル契約）

| 優先度 | 初期対応 | 完了期限 |
|--------|---------|---------|
| Critical | 1 時間 | 24 時間 |
| High | 4 時間 | 3 日 |
| Medium | 1 日 | 1 週間 |
| Low | 2 日 | 2 週間 |

---

## ✅ 運用チェックリスト

毎週チェック:

- [ ] 未対応の Critical がない
- [ ] SLA を守れている
- [ ] ユーザーからのリプライがある
- [ ] CHANGELOG が更新されている

毎月チェック:

- [ ] 統計が記録されている
- [ ] 改善トレンドがある
- [ ] チーム間の連携が取れている
